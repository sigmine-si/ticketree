## 구현 규칙 (내부)

목업 원본은 저장소 루트의 `client-portal-mockup.html`, `admin-dashboard-mockup.html`이다.
색·간격·아이콘은 그쪽이 기준이고, 여기서는 **무엇이 보이고 무엇을 누를 수 있는지**만
정한다. 화면을 새로 만들 때는 목업을 먼저 열어본다.

### 클라이언트 — 요청 목록 `/{프로젝트}/requests`

```
┌ 요청 │ 명세 ───────────────────────────────── 시 ┐
│ 요청 내역                          [+ 새 요청]    │
│ 클릭하면 대화와 견적을 확인할 수 있어요            │
│ ┌──────────┬──────────┬──────────────────────┐  │
│ │답변 필요 1건│진행 중 0건 │이번 달 확정 견적 ₩0  │  │
│ └──────────┴──────────┴──────────────────────┘  │
│ [전체 2] [답변 필요 1] [진행 중 0] [완료 1]        │
│ 번호  요청              진행       견적    업데이트 │
│  2   고객 포털 로그인   ●●●● 배포완료  50~75만  6시간 │
│  1   관리자 결정 알림   ●○○○ 답변필요  산출 중  7월22 │
└──────────────────────────────────────────────┘
```

- 상단 요약 세 칸이 먼저 온다. 그중 **답변 필요**가 클라이언트가 움직여야 하는 유일한 신호다.
- 진행은 4단계 세그먼트다 — 접수됨 · 개발 중 · 개발 완료 · 배포 완료.
  내부 status 12개는 여기서 한 줄 메타 문장으로만 새어 나온다("확인 질문이 도착했어요").
- 내 차례인 행은 앰버 점으로 표시한다.

### 클라이언트 — 요청 상세 `/{프로젝트}/requests/{id}`

타임라인이다. 위에서 아래로 쌓인다.

1. 내가 쓴 요청 원문(지금은 이래요 / 이렇게 바뀌면 좋겠어요)
2. 담당자 카드 — 확인 질문이 카드 안에 하나씩, 각각 답변 입력칸. `2/3 답변됨` 뱃지
3. 러프 견적 카드 — 금액 범위·기간·작업 범위
4. 확정 견적 카드 — 금액 하나와 **[견적 승인하고 진행]**
5. 진행 중이면 맨 아래에 회전 표시와 진행 문구 한 줄

- 버튼은 화면에 한 번에 하나만 산다. 지금 할 수 있는 일이 무엇인지 헷갈리면 안 된다.
- 대기 중에는 **문구가 바뀌어야 한다** — 30초 넘게 같은 문장이면 멈춘 것으로 보인다.

### 클라이언트 — 서비스 명세 `/{프로젝트}/spec`

왼쪽이 문서 목록(검색창 포함), 오른쪽이 본문이다.

- 목록은 제품 → 흐름 → 기능 순서로 층이 갈리고, 기능 사이에 구분선이 들어간다.
- 변경 예정이 있는 문서에는 앰버 점이 붙는다.
- 본문은 「이렇게 동작해요」 체크리스트가 먼저, 그 아래 「알려진 제약」, 마지막이 변경 이력이다.
- 완료는 체크(초록), 예정은 시계(앰버) + `예정 · REQ-014` 태그, 상태를 단정할 수 없는 줄은
  중립 점이다. **내부 섹션은 이 화면에 렌더하지 않고 검색에도 걸리지 않는다.**

### 관리자 — 검토 큐 `/admin`

첫 화면이 현황판이 아니라 **결정 목록**이다.

- 요약 네 칸: 검토 대기 · 실행 중 job · 오늘 AI 원가 · 이번 달 확정 견적
- 행마다 결정 종류 뱃지 — 답변 필요(빨강) · 명세 승인 · 배포 승인
- 상단 우측에 알림 종. 안 읽은 건수가 붙고, 누르면 목록이 열린다
- 우측 상단 **[프로젝트 붙이기]**

### 관리자 — 요청 상세 `/admin/{프로젝트}/requests/{id}`

왼쪽이 근거, 오른쪽이 결정이다.

- 왼쪽: 문답 기록 · 탐색 노트 · 명세 diff(추가 초록/삭제 빨강) · 코드 PR 링크
- 오른쪽: 견적 조정 입력칸, 그 아래 버튼 세 개 —
  **[Spec 승인 · 개발 시작]** / [명세 다시 쓰기] / [클라이언트에게 되묻기] / [반려]
- **버튼 아래에는 그 버튼이 일으키는 비가역적 동작을 한 줄로 적는다**
  ("승인하면 PR #12가 머지됩니다").
- 배포 단계에서 스키마가 바뀌었으면 경고가 뜨고, 반영을 확인하기 전에는
  [배포 완료로 표시]가 눌리지 않는다.

### 관리자 — 고객 계정 `/admin/invites`

계정별로 [초대 링크 발급]. 누르면 링크와 PIN이 **한 번만** 표시된다. 다시 볼 수 없고
재발급하면 이전 것이 무효가 된다는 사실을 그 자리에 적는다.

### 관리자 — 원가 `/admin/costs`

요약 네 칸(누적 원가·누적 견적·원가율·오늘) + 단계별 표 + 요청별 표. 원가가 큰 순서다.
금액 옆에는 항상 "환산가"라는 사실이 붙는다.

### 로그인 화면들

- `/invite/{토큰}` — 프로젝트 이름과 이름을 보여주고 PIN 입력칸 하나. 틀리면 남은 횟수를
  알려주고, 잠기면 재발급을 요청하라고 안내한다.
- `/admin/login` — 아이디·비밀번호.
- `/dev-login` — 로컬 전용. 계정을 눌러 바로 들어간다.

### 말투

- 클라이언트 화면의 모든 문장은 "~해요/~예요"다. 명령형을 쓰지 않는다.
- **"코드"라는 말을 쓰지 않는다.** 무엇을 뒤지는지가 아니라 무엇을 확인하는 중인지를 쓴다.
- 오류는 무엇이 잘못됐는지와 무엇을 하면 되는지를 함께 적는다.

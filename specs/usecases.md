# 유스케이스

버전: v1.0 · 마지막 변경 2026-07-24
순서: 3

## 이렇게 동작해요

- [x] 요청 하나가 접수부터 배포까지 가는 정상 경로가 정해져 있다
- [x] 중간에 막히거나 실패해도 요청이 사라지지 않고 사람에게 넘어간다
- [x] 사람이 답을 주면 처음부터가 아니라 막힌 지점부터 이어진다

## 알려진 제약

- 요청을 중간에 보류하거나 취소하는 길이 화면에 없다
- 요청 하나를 여러 개로 쪼개는 기능이 없다 (제안만 한다)

## 구현 규칙 (내부)

규칙만 나열하면 읽는 사람이 흐름을 머리로 합성해야 한다. 여기서는 **실제로 일어난
일**을 순서대로 적는다. 각 유스케이스 끝에 그때 어긋날 수 있는 것을 붙인다.

### UC-1. 요청 한 건이 배포까지 (정상 경로)

1. 클라이언트가 「새 요청」에 AS-IS/TO-BE를 적어 보낸다. 그 순간 `draft`가 생기고
   탐색 job이 큐에 오른다. 모달은 대화 화면으로 바뀐다.
2. 담당자가 저장소를 읽고 확인 질문을 최대 3개 보낸다. 클라이언트가 하나씩 답한다.
   **한 라운드의 질문이 전부 답변된 순간에만** 다음 라운드가 시작된다.
3. 더 물을 것이 없으면 변경 요청서와 러프 견적(범위)이 나온다. 클라이언트가
   「이 내용으로 요청하기」를 누르면 그때 REQ 번호가 붙는다.
4. 견적 산출 job이 확정 금액을 낸다. 관리자가 금액을 조정할 수 있다.
5. 클라이언트가 「견적 승인하고 진행」을 누른다. **여기가 첫 번째 게이트다.**
6. 명세 job이 `specs/`를 고쳐 명세 PR을 연다. 추가되는 항목은 `- [ ] (예정 · REQ-014)`다.
7. 관리자가 diff를 보고 승인한다. **두 번째 게이트다.** 러너가 PR을 머지한다.
8. 구현 job이 격리된 작업 공간에서 코드를 고치고 코드 PR을 연다.
9. 관리자가 배포를 승인한다. 러너가 머지한다. manual 어댑터는 여기까지다.
10. 운영자가 실제 배포를 마치고 「배포 완료로 표시」를 누른다. 명세의 예정 항목이
    `- [x]`가 되고 요청이 종결된다.

클라이언트에게는 이 열 단계가 **네 단계**로 보인다. 5번과 7번 사이(관리자 게이트)는
"개발 중"이다 — manual 배포라는 사실도, 관리자가 검토 중이라는 사실도 노출하지 않는다.

### UC-2. 왜 게이트가 둘인가

클라이언트가 답하는 질문과 관리자가 답하는 질문이 다르다.

- 클라이언트: **"이 내용과 이 금액에 동의하는가"** — 무엇을 원하는지는 클라이언트만 안다.
- 관리자: **"기술적으로 타당하고 에이전트를 풀어도 안전한가"** — 기존 동작이 깨질지,
  범위가 실제보다 큰지는 클라이언트가 알 수 없다.

하나로 합치면 둘 중 하나가 부실해진다. 클라이언트에게 기술 판단을 떠넘기거나,
관리자가 클라이언트 의사를 대신 정하게 된다.

**어긋날 수 있는 것** — 클라이언트가 승인했는데 관리자가 반려하면 요청이 `cancelled`로
플래그가 붙는다. 이때 클라이언트에게 무엇이라 설명할지는 (미정)이다.

### UC-3. 담당자가 막혔을 때 (에스컬레이션)

접수 대화 중에도, 구현 중에도 일어난다.

1. 에이전트가 명세만으로 판단할 수 없는 문제를 만난다. 억지로 진행하지 않고 멈춘다.
2. 요청에 `escalated` 플래그가 얹힌다. **stage 위치는 보존된다** — 클라이언트 화면의
   진행 표시는 그대로다.
3. 관리자 큐에 「답변 필요」로 뜬다. 관리자가 답을 적는다.
4. 그 답은 **우리 팀의 결정**으로 프롬프트에 실린다. 에이전트는 그대로 따르고
   클라이언트에게 되묻지 않는다.
5. 접수 중 막혔으면 대화가, 구현 중 막혔으면 구현이 재개된다. 처음부터 다시 하지 않는다.

**실제로 겪은 것** — 구현 중 막혔을 때 그때까지 한 작업을 통째로 버린 적이 있다.
백엔드를 정확히 구현하고 테스트까지 만들었는데 화면 코드가 저장소에 없어 멈춘
경우였다. 지금은 **변경이 있으면 blocker가 있어도 코드 PR로 보존한다.**

### UC-4. 명세가 잘못 쓰였을 때

클라이언트 답변이 부족한 게 아니라 우리 판단이 부족했던 경우다.

1. 관리자가 명세 PR을 보고 무엇이 잘못됐는지 적는다. 지침 없이는 다시 쓰기가 안 눌린다.
2. 러너가 열린 명세 PR을 닫고, 폐기된 버전 행을 `superseded`로 접는다.
3. 명세 job이 그 지침을 받아 다시 쓴다. **클라이언트에게는 아무것도 묻지 않는다.**

「클라이언트에게 되묻기」와 다르다. 그건 클라이언트만 답할 수 있는 것이 남았을 때다.

**실제로 겪은 것** — 첫 명세안이 로컬 개발 통로를 통째로 없애자고 했다. 그대로
구현했으면 개발이 막혔을 것이다. 지침을 주고 다시 쓰게 해서 "실제 서비스에서는
로그인 없이 들어갈 수 없다"로 계약을 좁혔다.

### UC-5. 새 프로젝트를 붙일 때

1. 관리자가 이름·주소·클라이언트·저장소를 적는다. 프로젝트는 `provisioning`으로 생긴다.
2. 온보딩 job이 저장소를 복제하고 코드를 읽어 명세 세 층과 규칙 문서 초안을 만든다.
3. PR이 열린다. **자동으로 머지하지 않는다** — 명세는 계약서라 우리가 읽지 않은 채로
   계약이 될 수 없다.
4. 사람이 고쳐 머지하면 그때부터 요청을 받는다.

### 비정상 경로

정상 경로보다 이쪽이 중요하다. 아래는 전부 실제로 겪은 것이다.

| 무슨 일 | 지금 어떻게 되는가 |
|---|---|
| job이 실패 | 1회 자동 재시도. 또 실패하면 `failed` 플래그가 얹히고 관리자 큐로 간다. **클라이언트 stage는 건드리지 않는다** — job 실패는 클라이언트가 풀 수 있는 문제가 아니다 |
| 러너가 죽거나 재시작 | 하트비트가 90초 멈추면 job이 큐로 돌아간다. 처음부터 다시 돈다 — 브랜치를 다시 따므로 재실행이 안전하다 |
| PR이 main 이동으로 충돌 | 머지 전에 브랜치를 최신 main에 맞춘다. 겹치지 않으면 자동으로 풀리고, **같은 줄 충돌이면 사람에게 넘긴다.** 러너가 임의로 한쪽을 고르면 승인된 것과 다른 게 머지된다 |
| 에이전트가 코드를 안 고침 | 구현 job이 실패로 끝난다. 변경이 없으면 PR을 만들지 않는다 |
| 에이전트가 엉뚱한 곳을 고침 | 커밋 경로가 `codePaths`로 제한돼 있어 그 밖은 버려진다. 에이전트가 git을 만졌어도 러너가 origin/main 기준으로 되돌린 뒤 다시 커밋한다 |
| 스키마가 바뀐 채 배포 | 배포 화면이 경고하고, 반영을 확인하기 전에는 완료로 표시할 수 없다 |
| 클라이언트가 대화 도중 창을 닫음 | draft로 남는다. **(미정) 모달 밖에서 이어갈 자리가 없다** — 실제로 요청 하나가 이렇게 멈춰 사람이 DB를 봐야 했다 |
| PIN을 5회 틀림 | 초대 링크가 잠긴다. 재시도로는 안 풀리고 관리자가 재발급해야 한다 |
| 클라이언트가 말한 현재 상태가 실제와 다름 | 그 차이가 1순위 확인 질문이 된다. 추측으로 진행하지 않는다 |
| 명세와 코드가 어긋난 것을 발견 | 이번 요청 범위면 고치고, 아니면 `discrepancies`에 적어 사람에게 넘긴다 |

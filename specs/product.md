# Ticket Tree

버전: v1.0 · 마지막 변경 2026-07-22

## 왜 만드는가

외주 개발에서 같은 사고가 반복된다.

**말로 오간 요구사항이 어디에도 안 남는다.** 카톡과 통화로 정하고, 두 달 뒤 "이렇게
하기로 했잖아요"가 부딪힌다. 근거로 꺼낼 문서가 없다. 있어도 착수 시점 기획서라
그동안 바뀐 것이 반영돼 있지 않다.

**견적에 근거가 없다.** 감으로 부르고, 틀리면 우리가 떠안거나 클라이언트가 불신한다.
실제로 얼마나 걸렸는지 쌓아두지 않으니 다음 견적도 감이다.

**클라이언트가 진행 상황을 모른다.** 물어봐야 알고, 물어보면 개발이 멈춘다.

**바뀐 걸 아무도 안 적는다.** 코드는 나갔는데 문서는 그대로다. 시간이 지나면 코드가
유일한 진실이 되고, 그때부터는 개발자만 답할 수 있는 서비스가 된다.

네 가지가 같은 뿌리에서 나온다 — **합의가 문서로 남지 않는다.** 그래서 이 도구는
합의를 남기는 자리를 파이프라인 한가운데 박아둔다. 명세를 고치지 않고는 코드를
고칠 수 없게 만든다.

## 무엇을 하는 물건인가

외주 개발을 **명세를 기준으로** 운영하는 도구다.

클라이언트가 "이렇게 바뀌면 좋겠어요"를 적어 보내면, 담당자가 실제 코드를 확인한 뒤
되물어 내용을 확정한다. 확정된 내용은 먼저 명세에 반영되고, 그 다음에 코드가 바뀐다.
클라이언트는 자기 서비스가 지금 어떻게 동작하기로 되어 있는지를 명세 화면에서 본다.

일하는 방식이 세 가지 원칙 위에 서 있다.

**명세가 계약서다.** 기능별 명세가 클라이언트와의 약속이자, 구현의 기준이자, 검수의
테스트 기준이다. 세 가지를 각각 다른 문서로 두지 않는다.

**모든 답은 코드를 본 뒤에 나온다.** 질문도 견적도 추측으로 만들지 않는다. 클라이언트가
말한 현재 상태와 실제 코드가 다르면, 그 차이가 가장 먼저 확인할 질문이 된다.

**사람이 게이트를 지킨다.** 클라이언트가 금액과 내용에 동의해야 개발이 시작되고,
관리자가 승인해야 배포된다. 되돌릴 수 없는 일은 반드시 사람의 승인 뒤에 일어난다.

## 누가 쓰나

- **클라이언트** — 요청을 내고, 질문에 답하고, 견적을 승인하고, 명세를 읽는다.
  자기 프로젝트만 본다.
- **관리자(우리 팀)** — 결정을 내린다. 담당자가 막힌 문제에 답하고, 명세 변경과 배포를
  승인한다. 프로젝트에 묶이지 않고 전부를 본다.

## 안 만드는 것

이건 **우리 팀이 우리 외주 프로젝트를 굴리는 내부 도구다.** 아래는 만들지 않는다.
필요해지면 그때 다시 정한다.

- 셀프 가입과 결제 — 프로젝트는 우리가 손으로 붙인다
- 여러 조직이 함께 쓰는 구조 — 조직 분리, 조직별 권한, 데이터 격리 보장
- 자동 알림 채널(이메일·슬랙)
- 정산과 청구서 — 금액은 기록만 한다
- 견적 자동 보정 — 데이터는 쌓되 모델은 만들지 않는다

| 단계 | 클라이언트에게 보이는 4개 — 접수됨 · 개발 중 · 개발 완료 · 배포 완료 |
| 담당자 | 클라이언트에게 보이는 호칭. 실제로는 에이전트다 |
| 워크스페이스 | 프로젝트 저장소의 클론. 명세와 코드가 여기 함께 있다 |

## 구현 규칙 (내부)

### 이 문서의 자리

명세는 세 층이다. **같은 사실을 두 곳에 적지 않는다** — 어긋나면 어느 쪽이 맞는지
알 수 없고, 실제로 어긋난 적이 있다(관리자 인증을 바꿨는데 전체 흐름 문서가 옛 내용을
유지했다). 사실마다 주인을 하나 정하고 나머지는 링크한다.

| 층 | 파일 | 주인인 사실 |
|---|---|---|
| 제품 | `specs/product.md` | 무엇을·누구를 위해·무엇을 안 하는가, 용어 |
| 흐름 | `specs/overview.md` | 어느 기능도 혼자 소유하지 않는 것 — 상태 목록, 레인 정책, job 순서, 큐와 락, 데이터 모델 |
| 기능 | `specs/features/*.md` | 그 기능의 계약과 구현 규칙, 그 기능이 수행하는 상태 전이 |

### 클라이언트 언어

클라이언트가 읽는 모든 문장에서 **"코드"라는 말을 쓰지 않는다.** 우리가 무엇을 뒤지는지가
아니라 무엇을 확인하는 중인지가 필요하다. 내부 status(12개)도 노출하지 않는다 —
4단계만 보여준다.

### 성공 기준

이 도구가 됐는지를 이걸로 판단한다.

- 요청 10건을 이 파이프라인만으로 처리한다 (도그푸딩 → 실 클라이언트 1곳)
- 견적 정확도·응답 소요·에스컬레이션 빈도를 요청마다 기록한다
- **명세만 읽고 서비스가 어떻게 도는지 알 수 있다.** 코드를 열어야 안다면 명세가 진 것이다

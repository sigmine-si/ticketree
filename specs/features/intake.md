# 요청 접수와 대화

버전: v1.0 · 마지막 변경 2026-07-22
순서: 1

## 이렇게 동작해요

- [x] 클라이언트는 "지금은 이래요 / 이렇게 바뀌면 좋겠어요"를 적어 새 요청을 낸다
- [x] 요청이 들어오면 담당자가 코드를 직접 확인한 뒤 되묻는다 — 추측으로 묻지 않는다
- [x] 질문은 한 번에 최대 3개까지만 오고, 답에 따라 갈리는 질문은 다음 차례로 미룬다
- [x] 클라이언트는 질문 하나씩 따로 답할 수 있고, 진행률("1/3 답변됨")이 표시된다
- [x] 한 번에 온 질문에 모두 답해야 다음 차례가 시작된다
- [x] 매번 아직 확정되지 않은 항목이 무엇인지 함께 알려준다
- [x] 클라이언트가 말한 현재 상태와 실제 동작이 다르면 그 차이를 먼저 확인한다
- [x] 더 물을 것이 없으면 변경 요청서와 대략적인 금액 범위가 나오고, 클라이언트가 확정한다
- [x] 담당자가 판단할 수 없는 문제를 만나면 클라이언트가 아니라 우리 쪽 관리자에게 넘어간다
- [x] 대화 진행 상황은 새로고침 없이 화면에서 바로 보인다

## 알려진 제약

- 파일 첨부는 아직 올릴 수 없다
- 요청이 여러 개로 쪼개져야 할 때 제안은 하지만, 실제로 쪼개주지는 않는다

## 구현 규칙 (내부)

### 흐름

`exploration`(첫 라운드, 새 세션) → `intake_round`(답변 후 `--resume`으로 재개) 반복.
두 job은 같은 코드를 지나고 차이는 resume 여부와 프롬프트뿐이다.

상태: `draft` → `exploring` → `awaiting_client` ⇄ `exploring` → (확정) `submitted`.
한 라운드의 결과는 `messages`(role=agent)에, 질문은 `message_questions`에 한 행씩 쌓인다.
에이전트 세션 id는 `agent_sessions`에 남아 다음 라운드가 대화 맥락을 복원한다.

에이전트 출력은 read-only 레인이라 파일로 쓸 수 없다. 최종 응답 안의 ```json 블록이
유일한 계약이다(outcome=questions | ready | escalate).

### 규칙

- 한 라운드의 질문은 최대 3개. 답에 따라 갈리는 조건부 질문은 다음 라운드로 미룬다.
- **한 라운드의 모든 질문이 답변된 순간에만** 다음 라운드 job이 큐에 등록된다.
  부분 답변으로는 에이전트를 깨우지 않는다 — 답이 덜 온 상태로 돌리면 같은 질문을 또 한다.
- 라운드 상한은 없다. 종료 조건은 "변경 요청서가 확정 가능한가" 하나뿐이다.
- 대화 레인은 Read·Grep·Glob만 쓴다. Bash는 일부러 뺐다 — 이 레인의 컨텍스트에는
  클라이언트가 쓴 문장이 들어오므로 임의 명령 실행을 열어줄 이유가 없다.
- effort는 medium 고정. low는 37~44초로 빠르지만 교차 파일 위험을 놓친다(실측).
  견적의 근거가 되는 발견이라 이 차이는 비용이 아니라 손실이다.
- 응답 지연은 탐색이 아니라 **생성**이 지배한다. 도구 호출 11회가 5초, 생성이 42초였다.
  출력 계약(JSON)에 필드를 더하면 그대로 느려진다.
- 첨부는 텍스트로 추출한 내용만 전달한다. 이미지는 "첨부 N건 있음"으로만 알린다 —
  프롬프트 주입 표면을 줄이기 위해서다.
- 에스컬레이션은 최후 수단이다. 클라이언트에게 물어서 진행할 수 있으면 질문이다.

# 개발과 배포

버전: v1.0 · 마지막 변경 2026-07-22
순서: 5

## 이렇게 동작해요

- [x] 명세가 승인되면 개발이 시작되고, 명세에 적힌 예정 항목이 그대로 수용 기준이 된다
- [x] 개발은 다른 요청과 격리된 작업 공간에서 이뤄져 서로 방해하지 않는다
- [x] 작업 결과는 검토 가능한 형태로 남고, 관리자 승인 전에는 반영되지 않는다
- [x] 개발 중에 명세만으로 판단할 수 없는 문제를 만나면 멈추고 관리자에게 넘긴다
- [x] 그렇게 멈춘 경우에도 그때까지 한 작업은 버리지 않고 남긴다
- [x] 관리자가 답을 주면 처음부터가 아니라 멈춘 지점부터 이어서 진행한다
- [x] 관리자가 배포를 승인하면 변경이 반영된다
- [x] 배포가 끝나면 요청이 종결되고 명세의 예정 항목이 정식 항목이 된다
- [x] 클라이언트에게는 접수됨 → 개발 중 → 개발 완료 → 배포 완료 네 단계만 보인다

## 알려진 제약

- 실제 배포는 사람이 직접 한다. 플랫폼은 반영까지만 하고 "배포 완료"는 관리자가 표시한다
- 미리보기 주소도 관리자가 직접 입력한다
- 자동 되돌리기는 없다

## 구현 규칙 (내부)

### 흐름

`implementation` → 코드 PR → 관리자 배포 승인 → `deploy`(머지) → 수동 배포 →
`deploy_finalize`(예정→정식) → 종결.

상태: `queued_dev` → `developing` → `in_review` → `awaiting_manual_deploy` → `deployed`.
코드 PR은 `pull_requests`(kind=code, repo_id=해당 repo)에 저장한다.
`deploy_finalize`가 명세의 `- [ ] (예정 · REQ-014)`를 `- [x]`로 바꿔 main에 올리고,
`spec_versions`를 live로 바꾼 뒤 `deployed_at`을 찍는다.

### 규칙

- 구현은 `work/{REQ}/` worktree에서 한다. main 체크아웃(명세 화면이 읽는다)과
  물리적으로 분리한다.
- 커밋 경로는 `projects.settings.codePaths`로 제한한다. 기본값은 `src`·`digest.md`,
  모노레포는 `packages`·`scripts` 등. 이 밖의 변경은 커밋되지 않고 버려진다.
- 에이전트가 중간에 git을 만졌어도 `reset --mixed origin/main`으로 되돌린 뒤
  러너가 결정적으로 다시 커밋한다. 브랜치명·커밋 메시지가 추적의 기준이라 그렇다.
- **변경이 있으면 blocker가 있어도 코드 PR로 보존한다.** 정확한 작업을 통째로
  버리지 않는다 — 사람이 이어받게 남긴다. (REQ-002에서 실제로 버렸던 적이 있다)
- 구현 중 에스컬레이션은 답변을 받으면 intake_round가 아니라 implementation을
  재개한다. 처음부터 다시 하지 않는다.
- 머지·배포는 러너만 한다. 에이전트가 닿는 코드 경로에 두지 않는다.
- manual 어댑터는 PR 머지까지만 하고 `awaiting_manual_deploy`로 둔다. 실제 배포와
  미리보기 주소는 운영자가 채운다. **클라이언트에게 manual이라는 사실은 노출하지 않는다.**
- 구현 레인은 Opus·effort high. 정확도가 속도보다 중요한 구간이라 여기서는 아끼지 않는다.
- worktree에는 node_modules가 없다. 타입체크가 필요하면 설치부터 해야 한다(1~2분).

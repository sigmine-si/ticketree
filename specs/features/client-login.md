# 고객 접속과 로그인

버전: v1.0 · 마지막 변경 2026-07-23
순서: 1

## 이렇게 동작해요

- [ ] (예정 · REQ-002) 관리자는 고객 계정마다 초대 링크와 PIN을 발급할 수 있다
- [ ] (예정 · REQ-002) 클라이언트는 전달받은 초대 링크로 들어가 PIN을 입력하면 로그인된다
- [ ] (예정 · REQ-002) PIN을 5회 연속 잘못 입력하면 그 초대 링크는 잠기고, 관리자가 다시
  발급해야만 풀린다
- [ ] (예정 · REQ-002) 실제 서비스에서는 로그인 없이 클라이언트 화면에 들어갈 수 없다

## 알려진 제약

- 초대 링크에는 만료 기한이 없다. 링크가 새어나가면 관리자가 다시 발급해서 무효화해야 한다
- 링크와 PIN은 관리자가 직접 전달한다. 문자·이메일 등 자동 발송은 하지 않는다(§11 정책)

## 구현 규칙 (내부)

### 흐름

관리자 화면에서 고객 계정별로 '초대 링크 생성'을 누르면 토큰과 6자리 숫자 PIN이
함께 발급된다. 저장은 해시만 한다(`users.inviteTokenHash`, `users.pinHash`) — 평문은
관리자 화면에 1회 노출되고, 그 뒤로는 다시 볼 수 없다.

공개 라우트 `/invite/[token]`이 토큰 유효성을 확인하고 PIN 입력 폼을 보여준다.
PIN이 맞으면 기존 세션 서명 레이어로 `setSession({kind:'client', projectId, userId})`를
발급한다. `requireClient()`를 그대로 재사용하므로 이후 화면(요청 목록·상세·명세)은
바뀌지 않는다.

### 규칙

- PIN은 숫자 6자리로 발급한다. 원문은 저장하지 않고 해시로만 보관한다
  (`pinHash`). 사유: 초대 링크가 새더라도 DB 유출만으로 PIN이 바로 드러나지
  않게 하려는 것이다.
- PIN 시도 제한은 **5회 연속 실패 시 잠금**이다. 잠긴 뒤에는 재시도로 풀리지
  않는다 — 관리자가 초대 링크와 PIN을 다시 발급해야만 그 계정으로 다시 들어갈 수
  있다(재발급은 기존 해시를 덮어써 이전 링크·PIN을 함께 무효화한다).
- (미정) 시도 횟수를 DB에 둘지 인메모리에 둘지 아직 정하지 않았다. 인메모리로
  두면 러너·웹 프로세스가 재시작될 때 시도 횟수가 초기화돼 그 순간 잠금이 풀린다 —
  배포 전 사람이 판단해야 한다.
- (미정) PIN 해시에 쓸 라이브러리를 아직 정하지 않았다. 현재 세션 서명은 HMAC만
  쓰고 있어 새 의존성이 추가될 수 있다.
- (미정) 초대 링크에 만료 기한을 두는 기능은 이번 범위 밖이다. 필요해지면
  별도 요청으로 다룬다. 그때까지는 재발급(해시 덮어쓰기)이 유일한 무효화 수단이다.
- **로컬 개발 전용 통로(`/dev-login`)는 남는다.** 이 화면은 로컬 개발에서 DB에
  로그인 흐름 없이 들어가는 유일한 문이라 통째로 없애면 개발이 막힌다. 대신
  관리자 로그인의 `devLogin`과 같은 방식으로 `NODE_ENV==='production'`일 때는
  막는다 — 실제 서비스에는 이 통로로 들어갈 수 없다.
- 관리자 로그인(아이디·비밀번호)은 이 변경의 범위가 아니다. [관리자 검토
  큐](admin.md)가 그대로 소유한다.
- 실제 로그인 흐름(초대 링크 발급 → PIN 입력 → 세션 발급 → 5회 잠금)은 데이터베이스가
  있어야 확인할 수 있어 이번 명세 반영 단계에서는 검증하지 못했다. 배포 전 사람이
  직접 확인해야 한다.

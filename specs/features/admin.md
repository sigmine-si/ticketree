# 관리자 검토 큐

버전: v1.1 · 마지막 변경 2026-07-22
순서: 5

## 이렇게 동작해요

- [x] 관리자 첫 화면은 현황판이 아니라 "지금 내가 내려야 하는 결정" 목록이다
- [x] 결정이 필요한 순서대로 정렬되고, 어떤 종류의 결정인지가 함께 표시된다
- [x] 결정 종류는 답변 필요 · 명세 승인 · 배포 승인 세 가지다
- [x] 지금 실행 중인 작업과 대기 중인 작업 수가 함께 보인다
- [x] 담당자가 멈춰 세운 문제는 관리자가 답을 주고 다시 진행시킬 수 있다
- [x] 관리자가 준 답은 우리 팀의 결정으로 취급되어 클라이언트에게 되묻지 않는다
- [x] 관리자는 아이디와 비밀번호로 로그인하고, 정해둔 한 계정만 들어올 수 있다
- [x] 모든 상태 변화가 기록되어 언제 무슨 일이 있었는지 되짚을 수 있다

## 알려진 제약

- 알림이 없다. 관리자가 화면을 직접 열어봐야 결정할 것이 있는지 안다
- 새 프로젝트를 붙이는 일은 아직 사람이 손으로 한다
- 요청을 보류하거나 취소하는 버튼이 화면에 없다

## 구현 규칙 (내부)

### 흐름

검토 큐는 테이블이 아니라 **파생값**이다. `change_requests`의 status·flag와
그 요청에 딸린 running job을 조합해 결정 종류를 뽑는다.

| 결정 | 언제 |
|---|---|
| answer | flag=escalated |
| spec | 명세 PR이 열려 있고 아직 승인 전 |
| deploy | status=in_review 또는 awaiting_manual_deploy |
| running / waiting / done | 사람이 할 일이 없는 상태 |

이 매핑은 DB를 안 쓰는 순수 모듈이라 클라이언트 컴포넌트도 그대로 import한다.

### 규칙

- 결정 종류는 status가 아니라 "지금 사람이 내려야 하는 결정"에서 나온다.
  answer(에스컬레이션 답변) · spec(명세 승인) · deploy(배포 승인) 셋이다.
- 이중 게이트: 클라이언트의 견적 승인과 관리자의 명세 승인은 다른 판단이라 둘 다 거친다.
  관리자 게이트 구간은 클라이언트에게 "개발 중"으로 보인다.
- 관리자가 준 답은 클라이언트의 말이 아니라 우리 팀의 결정으로 프롬프트에 실린다.
  에이전트는 그대로 따르고 클라이언트에게 되묻지 않는다.
- 관리자 인증은 `.env`의 ADMIN_ID / ADMIN_PASSWORD 한 쌍. 둘 다 상수시간으로 대조하고
  (아이디가 틀려도 비밀번호까지 비교한다 — 응답 시간으로 계정 존재가 새지 않게),
  통과 전에는 users 행조차 만들지 않는다. 클라이언트 세션과 관리자 세션은 별개 쿠키다.
- **(미정) 비밀번호가 .env에 평문으로 있고 계정이 하나뿐이다.** 관리자가 둘 이상 되거나
  외부에 열 때는 해시 저장 + users 테이블 기반으로 옮겨야 한다.
- 모든 상태 전이는 `transition`을 지나 request_events에 기록된다. 직접 UPDATE 하지 않는다 —
  클라이언트 타임라인·감사 추적·단계별 소요 분석이 전부 이 로그에서 나온다.
- job 실패는 1회 자동 재시도 후 에스컬레이션한다.
- **(미정) 러너가 죽으면 job이 running으로 남고 아무도 줍지 않는다.** 지금은 사람이
  DB에서 재등록해야 한다. 되찾는 장치가 필요하다.
- **(미정) 알림이 pending_notices에 쌓이기만 하고 읽는 화면이 없다.**

# 견적과 승인

버전: v1.0 · 마지막 변경 2026-07-22
순서: 2

## 이렇게 동작해요

- [x] 견적은 두 번 나온다 — 대화를 마칠 때의 대략적인 범위, 그 뒤의 확정 금액
- [x] 확정 견적에는 작업 분해와 예상 소요가 함께 나온다
- [x] 관리자는 청구 금액을 조정해서 확정할 수 있다
- [x] 클라이언트가 "이 내용과 가격에 동의"를 눌러야 개발이 시작된다
- [x] 클라이언트 승인만으로는 부족하다 — 관리자가 기술적으로 타당한지 한 번 더 본다
- [x] 두 승인 사이 구간은 클라이언트에게 "개발 중"으로 보인다
- [x] 작업에 실제로 들어간 AI 원가와 소요 시간이 요청마다 쌓인다
- [x] 관리자 첫 화면에서 오늘의 AI 원가와 이번 달 확정 견적 합계를 볼 수 있다

## 알려진 제약

- 표시되는 AI 원가는 환산가라서 실제 지출과 다르다 — 참고 지표로만 쓴다
- 과거 비슷한 규모의 요청이 얼마나 걸렸는지를 견적 화면에서 보여주지는 않는다
- 요청별·작업별로 원가를 파고들어 보는 화면은 없다
- 청구·정산 기능은 없다. 금액은 기록만 된다

## 구현 규칙 (내부)

### 흐름

티켓 확정 직후 `estimation` job이 돈다. 결과는 `estimates`에 version을 올려 쌓는다
(러프 견적은 접수 대화의 summary, 확정 견적은 이 job).

상태: `submitted` → `estimating` → `quote_ready` → (클라이언트 승인) `client_approved`.
관리자가 금액을 조정하면 새 version 행이 생긴다 — 덮어쓰지 않는다.

### 규칙

- 러프 견적은 범위(최소~최대)로, 확정 견적은 단일 금액으로 낸다.
- 확정 견적 job은 접수 대화 세션을 상속하지 않는다. 깨끗한 세션에서 확정된 변경
  요청서와 탐색 노트만 넘겨받는다 — 대화 맥락이 금액 판단을 흐린다.
- 견적에는 **검토·검수 시간을 반드시 포함한다.** 에이전트가 구현해도 사람이 읽고
  확인하는 시간은 든다.
- 금액은 원 단위 정수, 만원 단위 반올림.
- 원가는 job의 result 이벤트에서 total_cost_usd·modelUsage를 그대로 쌓는다.
  캐시 토큰(cache_creation + cache_read)을 입력에 합산한다 — 원가의 대부분이 여기서 나온다.
- 표시 환율은 고정값(USD_TO_KRW, 기본 1400)이다. 정산에 쓰지 않는다.
- 이 금액은 **API 환산가이지 구독 실지출이 아니다.** 원가율은 참고 지표로만 쓴다.
- 실측 기준선: 대화 한 라운드 약 $0.2~1.2, 구현 job 약 $0.4~1.1. 모노레포는
  입력 토큰이 38만~80만이라 단일 저장소보다 크다.

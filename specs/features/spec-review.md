# 명세 검토와 승인

버전: v1.0 · 마지막 변경 2026-07-22
순서: 4

## 이렇게 동작해요

- [x] 확정된 요청은 먼저 명세로 옮겨진다 — 코드보다 명세가 먼저 바뀐다
- [x] 명세는 클라이언트가 읽을 수 있는 말로 쓰인다
- [x] 관리자는 바뀐 부분을 이전 명세와 나란히 비교해 보고 승인한다
- [x] 승인해야 명세가 정식으로 반영되고, 그 다음에야 개발이 시작된다
- [x] 아직 배포되지 않은 항목은 "예정"으로 표시된다
- [x] 배포가 끝나면 예정 표시가 사라지고 정식 항목이 된다
- [x] 클라이언트는 우리 서비스가 지금 어떻게 동작하기로 되어 있는지를 명세 화면에서 본다
- [x] 명세에는 변경 이력이 남아 어느 요청으로 무엇이 바뀌었는지 알 수 있다

## 알려진 제약

- 명세 화면에서 옛 버전을 되돌려 보는 기능은 없다
- 명세와 실제 동작이 어긋난 것을 발견하면 기록은 남지만, 자동으로 잡아주지는 않는다

## 구현 규칙 (내부)

### 흐름

`spec_draft` → 명세 PR → 관리자 승인 → `spec_merge` → 개발 대기.

상태: `client_approved` → (명세 PR 열림) → 관리자 승인 → `queued_dev`.
`spec_versions`에 feature별로 proposed → merged → live 로 단계가 남고,
PR은 `pull_requests`(kind=spec, repo_id=NULL)에 diff까지 함께 저장한다 —
검토 화면이 GitHub을 다시 부르지 않게 하려는 것이다.

### 규칙

- 명세 한 장에 계약과 설계가 함께 있다. `## 이렇게 동작해요`가 계약,
  `## 구현 규칙 (내부)`가 설계다. 문서를 갈라두면 설계가 계약을 따라가지 못하고 썩는다.
- 화면에서 가른다. 제목에 '내부'·'구현 규칙'·'설계'가 들어간 섹션은 클라이언트
  화면에 렌더하지 않고, 검색 대상에서도 뺀다. 관리자는 명세 PR의 diff로 본다.
- 파서는 아는 형식만 통과시키지 않는다. 체크박스 없는 줄, 모르는 섹션도 살린다 —
  다만 체크박스 없는 줄은 완료로도 예정으로도 표시하지 않는다(중립 표시).
  실제 클라이언트 저장소를 온보딩하면 대부분 이 형태다.
- 머리말 `순서: N`으로 목록을 정렬한다. 없으면 가나다순으로 뒤에 붙는다.
- 배포 전 항목은 `- [ ] (예정 · REQ-014)`, 배포되면 `- [x]`. 이 치환은 결정적이라
  에이전트를 쓰지 않고 러너가 main에 직접 커밋한다.
- 명세 화면은 DB가 아니라 워크스페이스의 파일을 매 요청마다 읽는다. 진실이 Git이라
  그렇다 — 대신 머지 직후 syncMain을 부르지 않으면 화면이 옛 명세를 보여준다.
- 명세 PR 브랜치는 `spec/REQ-014`이고 `specs/`만 건드린다.
- **(미정) 개발 모드에서 내부 섹션 원문이 페이지 페이로드에 실린다.** 화면에는
  렌더되지 않지만 HTML의 RSC 페이로드에 파일 원문이 들어간다. pg 드라이버 내부
  객체(`_parsers`·`RowCtor`)까지 같이 실리는 것으로 보아 Next.js 개발 모드가 서버 IO를
  devtools용으로 싣는 것이다. 프로덕션 빌드가 깨져 있어 아직 확인하지 못했다 —
  **확인 전까지는 새는 것으로 취급한다.**

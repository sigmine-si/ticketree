# Ticket Tree

Spec Driven Development 기반 외주 개발 운영 플랫폼. 이 저장소는 플랫폼 자신의 코드이면서,
동시에 **플랫폼이 관리하는 프로젝트 중 하나**다 (도그푸딩 — spec.md §6, §16-8).

명세와 코드가 한 저장소에 있다. `specs/`가 계약, `packages/`가 구현이다.
`spec.md`는 착수 시점의 기획서(내부 문서)이고, `specs/`가 지금 유효한 명세다.
둘이 어긋나면 `specs/`가 이긴다.

## 플랫폼 규칙

- `specs/`가 유일한 진실이다. 코드가 명세와 다르면 그 차이를 보고하라.
- main 브랜치에 직접 push하지 않는다. 변경은 반드시 브랜치와 PR로 낸다.
- 명세와 요청이 충돌하면 추측해서 진행하지 말고 멈추고 보고하라.
- **PR을 머지하지 않는다.** 머지는 사람의 승인 뒤에 러너가 한다.
- 탐색 결과는 파일 경로와 함께 보고한다.

## 명세 작성 규칙

명세는 세 층이다. **같은 사실을 두 곳에 적지 않는다** — 어긋나면 어느 쪽이 맞는지
알 수 없다. 사실마다 주인을 하나 정하고 나머지는 링크한다.

| 층 | 파일 | 주인인 사실 |
|---|---|---|
| 제품 | `specs/product.md` | 무엇을·누구를 위해·무엇을 안 하는가, 용어 |
| 흐름 | `specs/overview.md` | 어느 기능도 혼자 소유하지 않는 것 — 상태 목록, 레인 정책, job 순서, 큐와 락, 데이터 모델 |
| 기능 | `specs/features/*.md` | 그 기능의 계약과 구현 규칙, 그 기능이 수행하는 상태 전이 |

기능 명세를 고치다가 흐름 문서에도 같은 사실이 적혀 있으면, 흐름 쪽을 **링크로 바꾼다.**
양쪽을 맞추지 말고 한쪽을 없앤다.

그리고 명세 한 장 안에서는 **계약과 설계가 함께** 있다. 섹션으로 가르고, 화면에서 가른다.

| 섹션 | 무엇 | 누가 보나 |
|---|---|---|
| `## 이렇게 동작해요` | 계약 — 클라이언트가 승인하는 문장 | 클라이언트 화면에 그대로 |
| `## 알려진 제약` | 지금 안 되는 것 | 클라이언트 화면에 그대로 |
| `## 구현 규칙 (내부)` | 설계 — 계산식·조건·경계값·미정 사항 | 관리자·구현 담당만 |
| `## 변경 이력` | 어느 요청으로 무엇이 바뀌었나 | 클라이언트 화면에 그대로 |

제목에 '내부'·'구현 규칙'·'설계'가 들어간 섹션은 `/spec`에서 렌더하지 않고 검색에도
안 걸린다. 문서를 두 개로 가르지 않는 이유는 하나다 — **갈라두면 설계가 계약을
따라가지 못하고 썩는다.**

- 계약은 클라이언트가 읽을 수 있는 말로 쓴다. 파일명·함수명·기술 용어를 쓰지 않는다.
- 계약은 "무엇이 일어나는가"로 쓴다. 코드가 어디 있는지는 digest.md의 몫이다.
- 설계에는 코드를 다시 뒤져야만 알 수 있는 것을 남긴다. 못 정한 것은 `(미정)`으로
  남긴다 — 비워두면 구현 담당이 혼자 정하게 된다.
- 배포 전 항목은 `- [ ] (예정 · REQ-014) ...` 형식으로 적는다. 배포되면 `- [x]`로 바꾼다.
- 머리말에 `순서: N`을 둔다. 명세 화면의 기능 목록이 이 순서로 정렬된다 —
  없으면 가나다순으로 밀려서 클라이언트가 끝단부터 읽게 된다.
- 변경 이력에는 날짜를 넣는다: `- v1.3 (REQ-014) 2026-07-22 설명`.
- `## 알려진 제약` 같은 다른 섹션도 클라이언트 화면에 그대로 나온다. 안 되는 것을
  숨기지 않는다 — 명세는 계약서라 제약이야말로 보여야 하는 쪽이다.

## 코드 규칙

- **상태는 DB, 진실은 Git, 러너는 일회용.** 웹앱과 러너는 직접 통신하지 않는다 —
  모든 상태는 DB를 거친다.
- 상태 전이는 `packages/shared/src/db/ops.ts`의 `transition`을 거친다.
  `change_requests`를 직접 UPDATE 하지 않는다 — 이벤트 로그가 끊긴다.
- 에이전트 실행은 `packages/runner/src/agent/claude.ts` 하나뿐이다. 다른 곳에서
  프로세스를 띄우지 않는다. 도구·권한·모델은 전부 `packages/shared/src/lanes.ts`의
  레인 정책에서 나온다 (§16-9).
- `--dangerously-skip-permissions`를 어디에도 두지 않는다.
- 머지·배포처럼 비가역적인 동작은 에이전트가 닿는 경로에 두지 않는다. 러너가 한다.
- 웹의 데이터 접근은 `packages/web/src/lib/data.ts`(클라이언트)와 `lib/admin.ts`(관리자)를
  거친다. 페이지·라우트가 drizzle을 직접 부르지 않는다 — 프로젝트 스코프를 빠뜨리게 된다.
- 클라이언트에게 보이는 문장에 내부 status를 노출하지 않는다. `packages/shared/src/status.ts`의
  4단계 stage만 보여준다.

## 확인 방법

worktree에는 `node_modules`가 없다. 타입체크가 필요하면 루트에서 `pnpm install` 후
`pnpm typecheck`를 돌린다(설치에 1~2분 걸린다). DB가 필요한 검증은 이 단계에서 하지 않는다 —
확인할 지점을 결과의 notes에 적어 사람에게 넘긴다.
